# Гибридная Blueprint-Консистентность для Flux.2 klein (исправлено)

**Исправленный план интеграции с учётом архитектуры Flux.2 (joint-attention)**  

## Общая концепция

Система создаёт компактный **Blueprint** объекта (персонажа, предмета) по нескольким его изображениям без обучения. На этапе генерации Blueprint‑токены **расширяют текстовый контекст** модели Flux, добавляя в него «визуально‑семантический отпечаток» объекта.  
Инъекция выполняется через стандартный механизм `patches` ComfyUI, **не трогая веса модели** и **не требуя текстового энкодера на каждом инжекторе**.

### Почему старый план не работал?

Flux.2 использует **единое (joint) внимание** без раздельных cross/self‑attention. Методы `set_model_attn2_patch` / `set_model_attn1_patch` не имеют аналогов во внутренних блоках.  
Вместо этого мы будем **расширять тензор `txt` на входе в блоки внимания** через хук `post_input`, который уже присутствует в официальном коде Flux.

## 1. Нода Blueprint Creator (без изменений)

Работает так же, как и раньше. Кратко:
- По нескольким видам объекта создаётся `z_hyb` (гибридный вектор) и опциональный `keyword_embedding` (токен имени).
- Сохраняется в `.blueprint` (safetensors) и индексный `.index.json`.

*Подробнее см. в `plan.md` (секция 1) – всё остаётся актуальным.*

## 2. Нода Blueprint Injector (исправленная)

### 2.1 Входы

| Параметр | Тип | Описание |
|----------|-----|----------|
| `model` | `MODEL` | Модель Flux.2 (клон ModelPatcher) |
| `blueprint_path` | `STRING` | Путь к `.index.json` (автоматически берётся последний `.blueprint`) |
| `blueprint_scale` | `FLOAT` (default 1.0) | Масштаб для blueprint‑токенов и keyword |
| `K_cross` | `INT` (default 16) | Сколько blueprint‑токенов генерировать |
| `seed_W` | `INT` (фикс.) | Seed матрицы `W_cross` |
| `inject_self_attn` | `BOOLEAN` (default False) | **Временно отключено** – не реализовано для Flux.2 (будет в будущем) |
| `self_attn_scale` | `FLOAT` | (не используется пока) |
| `bp_tokens_position` | `COMBO` (`append`, `prepend`) | куда добавлять Blueprint‑токены относительно text‑токенов |
| `keyword_position` | `COMBO` (`after_text`, `before_text`, `disabled`) | позиция keyword‑токена |

> **Текстовый энкодер и prompt сцены исключены из Injector** – промпт подаётся через отдельную `CLIPTextEncode`, как обычно.

### 2.2 Выход

- `model` (`MODEL`) – модель с применённым патчем `post_input`.

### 2.3 Внутренняя логика

1. **Загрузка blueprint**  
   Из `.blueprint` файла читаются:
   - `z_hyb` (вектор длиной `d_hyb`, обычно 1172)
   - `keyword_embedding` (опциональный, [D_qwen] или [d_ctx], если предварительно спроецирован)

2. **Lazy инициализация матриц**  
   При первом вызове патча (во время генерации) получаем `d_ctx = txt.shape[-1]`.  
   По размерам `d_hyb` и `d_ctx` создаём фиксированную матрицу `W_cross` [K_cross * d_ctx, d_hyb] (с seed).  
   Если `keyword_embedding` задан и его размерность ≠ `d_ctx`, создаём матрицу `W_kw` [d_ctx, D_kw].

3. **Функция патча `post_input`**  
   Добавляется в `transformer_options["patches"]["post_input"]`.  
   На каждом шаге:
   - Вычисляет `T_bp = (W_cross @ z_hyb).reshape(1, K_cross, d_ctx) * blueprint_scale`.
   - Берёт `keyword_token = (keyword_embedding @ W_kw).unsqueeze(0).unsqueeze(0) * blueprint_scale` (если есть keyword).
   - Формирует итоговый текстовый контекст `txt_new`:
        * Базовый порядок: `[txt, keyword_token, T_bp]` или `[keyword_token, T_bp, txt]` в зависимости от параметров.
   - **Важно: обновляет `txt_ids`**, добавляя для каждого нового токена корректные позиционные индексы (продолжая linspace после конца оригинальных текстовых индексов).
   - Возвращает обновлённые `img`, `txt`, `img_ids`, `txt_ids`.

4. **Применение к модели**  
   - `m = model.clone()`
   - В `m.model_options["transformer_options"]` добавляется (или модифицируется) ключ `patches` с функцией пост‑инпута.
   - Модель готова к дальнейшей обработке.

### 2.4 Схема работы в ComfyUI

```
Load Checkpoint (Flux.2)
   → CLIP Text Encode (промпт сцены) → [текстовые токены]
   → BlueprintInjector (персонаж A)   → [добавляет токены A]
   → BlueprintInjector (персонаж B)   → [добавляет токены B]
   → Sampler
   → VAE Decode
```

Каждый инжектор расширяет `txt` новыми токенами, не мешая друг другу.

## 3. Преимущества исправленного подхода

- **Полная совместимость с Flux.2** – используем штатные `patches`, ни один внутренний слой не изменён.
- **Не нужен text_encoder на каждой ноде** – промпт сцены задаётся один раз отдельно.
- **Мультиперсонажные сцены** – подключаем несколько инжекторов последовательно, каждый добавляет свои токены.
- **Не требует обучения** – только фиксированные случайные проекции.
- **Keyword‑привязка** – опциональный токен имени гарантирует, что даже без упоминания в промпте объект появится под правильным именем.
