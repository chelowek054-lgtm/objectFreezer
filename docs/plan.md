Вся система Blueprint-консистентности естественно разбивается на два независимых этапа, поэтому и в ComfyUI потребуется две основные кастомные ноды (и один служебный модуль):

1. **Blueprint Creator** – создаёт `.blueprint`-файл из набора изображений объекта.  
2. **Blueprint Injector** – на этапе генерации считывает blueprint и дополняет контекст внимания Flux.2 klein, обеспечивая консистентность.

Ниже – полный архитектурный разбор без кода.

---

## Общая диаграмма потоков (высокоуровнево)

```
[Папка с N изображений] ─┐
[Метаданные (ракурс,...)]─┤
                          ▼
               ┌─────────────────────┐
               │  Blueprint Creator  │
               │  (новая нода)       │
               └─────────┬───────────┘
                         │ сохраняет .blueprint (файл)
                         ▼
                 (файловая система)
                         │
                         ▼
               ┌─────────────────────┐
[Промпт сцены]─┤  Blueprint Injector ├──► [VAE decode] ──► изображение
               └─────────────────────┘
                   (внедряется в cross/self-attention Flux.2 klein)
```

---

## 1. Нода Blueprint Creator

**Назначение:** по нескольким видам объекта построить сжатое гибридное представление и сохранить его в `.blueprint` (формат safetensors).  
**Категория в ComfyUI:** `"blueprint"` (или `"object_consistency"`).

### Входы

| Параметр | Тип | Описание |
|----------|-----|----------|
| `images` | `IMAGE` (батч) | N кадров объекта (минимум 1, рекомендуется ≥3). |
| `vae` | `VAE` | Flux2‑VAE (модель, совместимая с Flux.2). |
| `text_encoder` | `CLIP` | Qwen (CLIP-like text encoder), используется `tokenize()` + `encode_from_tokens()`. |
| `ollama_url` | `STRING` | URL Ollama, используется `POST {ollama_url}/api/chat`. |
| `vlm_model` | `STRING` | Имя vision-модели в Ollama (например `qwen3-vl`). |
| `object_id` | `STRING` | Уникальное имя объекта (по умолчанию `"object"`). |
| `object_class` | `COMBO` (`["character","building","prop"]`) | Класс объекта, влияет на `z_face` (для не-character → нули). |
| `output_dir` *опц.* | `STRING` | Папка для сохранения blueprint (пусто → `ComfyUI/output/blueprints`). |
| `seed` *опц.* | `INT` | Seed для фиксированных матриц `R_sem`/`R_face` (создаются 1 раз и кешируются на диске). |

### Выходы

| Параметр | Тип | Описание |
|----------|-----|----------|
| `blueprint_path` | `STRING` | Путь к созданному `.blueprint`‑файлу. |

### Внутренняя логика (FUNCTION `create_blueprint`)

1. **Визуальный эмбеддинг:**  
   - Каждое изображение кодируется VAE в латент `l_i`.  
   - Применяется глобальный средний пулинг → вектор `v_i`.  
   - Усреднение по всем N → `z_vision` (размер 16).

2. **Семантический эмбеддинг:**  
   - Изображения отправляются в VLM (через HTTP‑запрос к API или локальный сервер).  
   - VLM возвращает `summary_text` (2‑3 предложения) и JSON‑словарь с деталями (материалы, пропорции, стиль).  
   - Текст `summary_text` кодируется Qwen → последовательность скрытых состояний.  
   - С помощью **фиксированной случайной матрицы** `R_sem` (генерируется один раз и сохраняется в папке плагина) множественные токены сворачиваются в один вектор `z_sem` размерности 1024.

3. **Геометрический эмбеддинг:**  
   - Из JSON-деталей извлекаются числовые пропорции (рост, ширина плеч и т.п.) – вектор `z_geo` размерности 20 (дополняется нулями).

4. **Лицевой эмбеддинг (опционально):**  
   - Если `class == "character"`, VLM дополнительно генерирует детальное описание лица; текст кодируется Qwen и проецируется второй фиксированной матрицей `R_face` → `z_face` (512).  
   - Для других классов – нулевой вектор.

5. **Сборка гибридного вектора:**  
   `z_hyb = concat([z_vision, z_sem, z_geo, z_face])` (1572 элемента).

6. **Сохранение `.blueprint`:**  
   - Все векторы (`z_vision`, `z_sem`, `z_geo`, `z_face`, `z_hyb`) и текстовая мета-информация пакуются в safetensors‑контейнер.  
   - Матрицы проекции `W_cross`, `W_self` в сам файл blueprint **не** сохраняются – они являются частью системного конфига и загружаются нодой Injector.

**Важно:** в процессе создания blueprint не требуется никакого обучения или обновления весов – используются только замороженные модели и однократные случайные проекции.

### Контракт Ollama `/api/chat` (как используется)
- Отправляется `messages=[system,user]`, где `user` содержит `images` (base64 PNG) и инструкцию вернуть **строгий JSON** со схемой:
  - `summary_text`: string
  - `details`: object (по возможности с числовыми пропорциями)
  - `face_description`: string|null (null если не `character`)

## 2. Нода Blueprint Injector

**Назначение:** встроить гибридный эмбеддинг в слои внимания денойзера Flux.2 klein прямо во время инференса.  
**Категория:** `"blueprint"`.

### Входы

| Параметр | Тип | Описание |
|----------|-----|----------|
| `model` | `MODEL` | Пайплайн Flux.2 klein (обёртка модели диффузии с трансформером). |
| `blueprint_path` | `STRING` | Путь к файлу `.blueprint`. |
| `scene_prompt` | `STRING` | Текстовое описание сцены (позитив). |
| `text_encoder` | `CLIP` | Qwen (CLIP-like) text encoder. |
| `blueprint_scale` | `FLOAT` (default 1.0) | Множитель амплитуды Blueprint‑токенов (`α`). |
| `inject_self_attn` | `BOOLEAN` (default False) | Включать инъекцию в self‑attention (`attn1_patch`). |
| `self_attn_scale` *опц.* | `FLOAT` (default 0.5) | Доп. масштаб для self‑attention (умножается на `blueprint_scale`). |
| `K_cross` *опц.* | `INT` (default 16) | Кол-во blueprint-токенов в cross-attention. |
| `M_self` *опц.* | `INT` (default 0) | Кол-во токенов для self-attn (если 0 и флаг включён — используется 8). |
| `seed_W` *опц.* | `INT` | Seed фиксированных матриц `W_cross/W_self` (кешируются в `custom_nodes/object-freezer/matrices/`). |
| `bp_tokens_position` *опц.* | `COMBO` (`append|prepend`) | Куда добавлять blueprint-токены относительно текстового контекста. |

### Выходы

| Параметр | Тип | Описание |
|----------|-----|----------|
| `model` | `MODEL` | Модифицированная модель, готовая к сэмплированию (с перехваченными слоями внимания). |

### Внутренняя логика (FUNCTION `inject`)

1. **Загрузка Blueprint:**  
   - Читает `z_hyb` из safetensors (`.blueprint`) или из debug JSON (`.blueprint.json`, берётся последняя запись `entries[-1]`).  
   - Матрицы `W_cross/W_self` генерируются детерминированно и кешируются в `custom_nodes/object-freezer/matrices/`.

2. **Формирование Blueprint‑токенов:**  
   - `T_bp = reshape(W_cross @ z_hyb)` → K токенов размерности `d_cross` (например, 16×2048).  
   - При `inject_self_attn=True`: `T_self = reshape(W_self @ z_hyb)` → M токенов размерности латентного патча.

3. **Кодирование текста сцены:**  
   - `text_tokens = text_encoder.encode(scene_prompt)` → последовательность токенов от Qwen.  
   - (Аналогично можно закодировать `negative_prompt` для CFG.)

4. **Модификация модели (хукинг внимания):**  
   Нода патчит модель через `ModelPatcher` patches:
   - **В cross‑attention:**  
     Контекст (ключи‑значения) расширяется конкатенацией `[text_context; T_bp * α]` через `attn2_patch`.  
   - **В self‑attention (опционально):**  
     В self-attn `attn1_patch` расширяются K/V токенами `T_self * β`.

   Эта подмена делается без изменения весов — только через patches ComfyUI (`set_model_attn2_patch`, `set_model_attn1_patch`).

5. **Возврат модифицированной модели:**  
   Другие ноды (Sampler, VAE Decode) работают дальше штатно, не подозревая о вмешательстве. Модель теперь при каждой генерации будет дополнительно кондиционироваться blueprint.

### Нюансы архитектуры
- Матрицы `W_cross`/`W_self` можно хранить в отдельной папке плагина (`blueprint_matrices/`) или вшивать в код при инициализации. Их не нужно менять, но пользователь при желании может перегенерировать seed.
- Для поддержки разных версий Flux/diffusion‑моделей на вход может приниматься указание типа слоёв (например, Flux‑transformer).  
- `blueprint_scale` управляет силой встраивания: 0 – объект не влияет, >1 – гипер‑усиление свойств объекта.

## 3. Опциональная нода: Blueprint Editor

Позволяет редактировать семантическую часть blueprint без пересоздания визуальной.

**Входы:** путь к blueprint, новый текст описания, text_encoder.  
**Выходы:** новый `.blueprint` с обновлёнными `z_sem`, `z_face`.  

Полезна для тонкой настройки “на лету” (изменение материала, стиля).

---

## Интеграция в пайплайн ComfyUI (пример схемы)

```
Load Image Batched → Blueprint Creator → (сохраняется файл)
...
Load Checkpoint (Flux.2) → CLIP Text Encode (промпт) → Blueprint Injector 
       └→ Sampler → VAE Decode → Preview Image
```

Нода Injector располагается **до** Sampler, модифицирует модель, и уже модифицированная модель передаётся в Sampler.

В силу архитектуры ComfyUI, где каждая нода может вернуть `MODEL`, такой подход абсолютно корректен.

---

## Что нужно реализовать в коде (тезисно)

- Два класса нод: `BlueprintCreator`, `BlueprintInjector`.  
- В `BlueprintCreator` – сериализация словарей и тензоров в safetensors (можно использовать библиотеку `safetensors`).  
- В `BlueprintCreator` – HTTP‑клиент для API VLM (или вызов локальной модели).  
- В `BlueprintInjector` – создание хуков на слои внимания внутри стандартной модели Flux (поиск по имени слоя, содержащему `cross_attn`/`self_attn`).  
- Фиксированные случайные матрицы создаются один раз при первом запуске и сохраняются в зависимости (можно использовать `numpy.random.RandomState` с фиксированным seed).  

Оба узла полностью соответствуют шаблону из `freezer.py`: категория, `INPUT_TYPES`, `RETURN_TYPES`, плюс может быть `FUNCTION`.  

Таким образом, вся система Blueprint-консистентности естественно вписывается в модульную архитектуру ComfyUI, не требует обучения и не конфликтует с другими расширениями.
